# Aviation Logistics Platform

## Context

A Singapore-based aviation software house needed to move an internal logistics and scheduling suite off a tightly-coupled WCF + WebForms stack toward something their airline customers could actually use from modern browsers, on spotty networks, with high transaction volumes during peak hours.

## Problem

Scheduling resources across airlines — crews, ground equipment, cargo — is inherently transactional: every change ripples. The existing system had two main pain points:

- The WebForms UI was slow to iterate on, and front-line ops teams disliked the workflow.
- Integrations with airline back-offices went through WCF services that were not idempotent, so retries during network blips caused double-bookings.

## Approach

- **Frontend**: Rebuilt the primary scheduling screens as an Angular SPA backed by a focused Web API. Kept round-trips small and optimistic updates explicit.
- **Services**: Introduced idempotency keys and structured retry semantics around WCF endpoints so the API layer could safely handle flaky downstream calls.
- **Collaboration**: Standup, backlog grooming, and sprint review across Cebu and Singapore — written handoffs in Confluence to keep the async pipeline healthy.

## Outcome

The platform was adopted across multiple airline customers. The scheduling surface got faster to use and harder to misuse, and the integration layer got predictable enough that support tickets around "phantom bookings" dropped off the board.

## Stack notes

- **Angular** (forms, reactive streams for live schedule updates)
- **ASP.NET Web API** as a thin integration layer
- **WCF** for legacy service contracts
- **SQL Server** for the system of record
